# novel-sync：项目维护与复盘

用法示例：
```
/novel-sync                      ← 全量维护（推荐每10章跑一次）
/novel-sync --fix-log            ← 只修复情节流水账
/novel-sync --arc-check          ← 只检查大纲偏移
/novel-sync --refresh-style      ← 只做 Reader反馈→文风复盘
/novel-sync --refresh-blurb      ← 只更新发布简介
/novel-sync --refresh-cover-refs ← 只刷新竞品封面参考
```

---

## Phase 0：加载项目状态

读取以下文件，建立全局状态：
```
novels/[小说名]/情节流水账.md
novels/[小说名]/大纲.md
novels/[小说名]/我的文风.md
novels/[小说名]/发布信息.md
novels/[小说名]/chapters/（所有文件列表 + 修改时间）
```

统计：
- 已写章节数（chapters/ 文件数）
- 流水账记录数
- 缺口章节列表（有文件但流水账无记录）

---

## Phase 1：情节流水账断档修复（`--fix-log`）

**1.1 找缺口**

对比 `chapters/` 目录文件与 `情节流水账.md` 中的记录：
```bash
# 从文件名提取章节号列表
ls novels/[小说名]/chapters/ | grep -oP 'ch\d+' | sort

# 从流水账提取已记录章节号
grep -oP '\[ch\d+\]' novels/[小说名]/情节流水账.md | sort
```

找出"有文件但流水账无记录"的章节。

**1.2 补充缺失记录**

对每个缺失章节，启动 **general-purpose agent** 读取章节文件，生成标准流水账条目：

```
[chXXX] [标题]
- 主要事件：[2-3句话概括]
- 新出场人物：[名字，或"无"]
- 情节转折：[如有，否则"无"]
- 结尾状态/钩子：[最后一幕 + 引出下章的悬念]
```

将缺失条目按章节号顺序**插入**到流水账正确位置（不是追加到末尾）。

**完成报告**：
```
✓ 修复流水账：补充了 [N] 条缺失记录
  补充章节：ch[X], ch[X], ch[X]
```

---

## Phase 2：大纲偏移检查（`--arc-check`）

启动 **general-purpose agent** 对比分析：

**输入**：
- `大纲.md`（按弧段分解的节点列表）
- `情节流水账.md`（所有已写章节的摘要）

**分析任务**：
1. 逐个大纲节点，判断是否已在流水账中出现
2. 识别实际写作中出现的重要情节，判断是否在大纲中有对应节点
3. 当前写作进度处于大纲哪个位置

**输出格式**：
```
══════════════════════════════
大纲对齐报告
══════════════════════════════
当前进度：ch[NNN]（共写了[N]章）
大纲位置：[当前处于哪个弧段的哪个节点]

已完成大纲节点（[M]/[总节点数]）：
  ✓ [节点描述]
  ✓ [节点描述]

未触及大纲节点（计划中）：
  · [节点描述]（预计 ch[XXX] 左右）

⚠ 偏移提醒：
  · [实际出现但大纲未规划的情节/人物]
  · [大纲有但已明显跳过/延迟的节点]

建议：[是否需要更新大纲 / 是否偏离严重]
```

若偏移严重，询问用户是否需要调用 `/novel-arc` 更新大纲。

---

## Phase 3：Reader反馈 → 文风复盘（`--refresh-style`）

**3.1 对最近10章跑 Reader 评分**

按章节号取最近10章文件，启动 **general-purpose agent** 以目标读者身份批量评估：
- 读取 `读者画像.md`
- 逐章打分，记录每章的：得分 / 最出戏的地方 / 钩子是否有效

**3.2 提取高频问题**

统计10章中反复出现的问题（出现≥3次视为模式）：

常见模式类别：
- 开头套路化（如总是以环境描写开场）
- 钩子类型单一（如总是"悬念"，缺乏变化）
- 对话比例失衡（过多或过少）
- 打斗/修炼场景节奏拖沓
- 心理描写过多
- 某类词汇/句式反复出现

**3.3 更新我的文风.md**

启动 **general-purpose agent**，将发现的模式以"改进规则"的形式追加到 `我的文风.md`：

```markdown
## 基于 Reader 复盘的写作改进（[YYYY-MM-DD]）

以下规则来自 ch[X]-ch[X] 的 Reader 评分分析：

- [问题描述]：[具体改进方向]
- [问题描述]：[具体改进方向]

平均 Reader 分：[X.X] / 10（上次复盘：[X.X]）
```

告知用户已更新，并展示新增的规则列表。

---

## Phase 4：发布简介迭代（`--refresh-blurb`）

读取：
- `发布信息.md`（当前简介）
- `情节流水账.md`（已写内容摘要）

启动 **general-purpose agent** 生成更新后的简介：
- 基于已写内容提炼真实的故事钩子
- 保持100-300字，克制有力，不像广告文案
- 保留原简介的调性，只强化细节和准确性

用 `AskUserQuestion` 展示新旧简介对比，询问是否采用：
```
当前简介：
[原文]

建议更新为：
[新版本]

采用新版本？Y/N（或"改这里：..."）
```

用户确认后写入 `发布信息.md`。

---

## Phase 5：平台数据提醒

检查最早一批已发布章节的发布时间（从章节文件修改时间推算）：
- 发布超过7天 → 提醒："建议去番茄小说后台查看前几章的阅读数据，判断留存情况"
- 发布不足7天 → 跳过，注明"[N]天后建议查看数据"

输出提醒（不自动打开浏览器，只提醒）：
```
📊 数据提醒：
  ch001-ch010 已发布 [N] 天
  建议在番茄小说后台查看：阅读量趋势 / 第几章掉量最明显
  数据入口：番茄小说创作中心 → 数据分析
```

---

## Phase 6：竞品封面参考刷新（`--refresh-cover-refs`）

```bash
# 删除旧的参考目录，下次运行 /novel-cover 时会自动重新抓取
rm -rf novels/[小说名]/封面参考/
```

告知用户：
```
✓ 封面参考已清除
  下次运行 /novel-cover 时会自动抓取最新排行榜封面
```

---

## 全量运行顺序

```
Phase 1（修复流水账）
  ↓
Phase 2（大纲对齐检查）
  ↓
Phase 3（Reader复盘 → 文风更新）    ← 最耗时，但最有价值
  ↓
Phase 4（简介更新，需用户确认）
  ↓
Phase 5（数据提醒，仅输出）
  ↓
汇总报告
```

## 汇总报告

```
══════════════════════════════
novel-sync 维护报告 [YYYY-MM-DD]
══════════════════════════════
小说：[书名] | 当前进度：ch[NNN]

[1] 流水账修复：补充 [N] 条 / 无缺口 ✓
[2] 大纲对齐：进度 [M]/[总] 节点 / ⚠ 有偏移（见上方详情）
[3] 文风复盘：发现 [N] 个高频问题，已更新我的文风.md
    Reader 平均分趋势：[上次X.X → 本次X.X]
[4] 发布简介：[已更新 / 用户跳过 / 无需更新]
[5] 平台数据：[已提醒 / 暂不需要]
[6] 封面参考：[已刷新 / 未执行]
```

## 运行建议

| 时机 | 推荐命令 |
|------|---------|
| 每写完10章 | `/novel-sync`（全量） |
| 章节文件和流水账对不上时 | `/novel-sync --fix-log` |
| 感觉故事跑偏了 | `/novel-sync --arc-check` |
| Reader分数持续偏低 | `/novel-sync --refresh-style` |
| 准备新一轮发布前 | `/novel-sync --refresh-blurb` |
